# 摘要

## 中文摘要

藏红花具有较高的药用价值和经济价值，但其生长过程对温度、湿度、光照和土壤水分等环境条件极为敏感。传统培育方式长期依赖人工巡检和经验判断，存在监测不连续、响应滞后、劳动强度大以及异常情况处置不及时等问题。针对这些痛点，本文设计并实现了一套面向藏红花全生长周期的边缘计算智能培育系统。

系统采用“感知与执行层—边缘计算层—应用层”的三层结构。感知与执行层以 STM32F411 为核心，接入 DHT11、BH1750、电容式土壤湿度计以及 PAJ7620 手势识别模块，实现环境数据采集、现场显示与执行控制；边缘计算层以树莓派 4B 为核心节点，部署 Flask Web 服务、SQLite 数据库和自动灌溉守护线程，将关键策略与控制逻辑下沉至本地；应用层通过响应式 Web 界面完成数据展示、历史查询、参数配置和远程控制，并结合 OpenCV 与本地部署的 Qwen 量化模型，实现花期识别与种植建议生成。

测试结果表明，系统能够稳定完成环境监测、历史数据存储、手势交互、自动灌溉和本地智能分析等功能。在弱网或断网条件下，边缘端仍可依据本地策略完成灌溉控制，体现出较好的实时性、可靠性和自治能力。本文的研究为名贵中药材的数字化、精细化和低成本智能培育提供了可实施的软硬件协同方案。

**关键词：** 边缘计算；藏红花；物联网；STM32；自动灌溉；机器视觉

## Abstract

Saffron is a high-value medicinal crop whose growth is highly sensitive to temperature, humidity, illumination, and soil moisture. Conventional cultivation still relies heavily on manual inspection and experience, leading to discontinuous monitoring, delayed control, and high labor cost. To address these issues, this paper designs and implements an edge-computing-based intelligent cultivation system for the full growth cycle of saffron.

The system adopts a three-layer architecture including the perception and execution layer, the edge computing layer, and the application layer. STM32F411 is used at the device side to collect environmental data and drive actuators. Raspberry Pi 4B serves as the edge node to run Flask services, SQLite database, and local irrigation logic. A responsive Web platform provides monitoring, control, and historical query functions. OpenCV and a locally deployed quantized Qwen model are further integrated for flower-stage recognition and cultivation advice.

Experimental results show that the system can continuously support environmental monitoring, local decision-making, unattended irrigation, and intelligent assistance. Even under weak-network or offline conditions, the edge side can still maintain key control functions. The proposed work provides a practical solution for digital and refined cultivation of valuable medicinal crops.

**Keywords:** Edge Computing; Saffron; Internet of Things; STM32; Automated Irrigation; Machine Vision

# 第一章 绪论

## 1.1 研究背景与意义

藏红花被誉为“红色金子”，兼具较高的经济价值和药用价值，但其培育过程对环境条件要求严格。球茎休眠、花芽分化、开花和营养生长等阶段对温度、湿度、光照和土壤水分的需求并不一致，稍有偏差就可能导致花期缩短、球茎腐烂或产量下降。传统种植方式主要依赖人工巡检和经验调控，存在监测间断、控制滞后、劳动成本高和可复制性差等不足。

随着物联网技术的发展，农业环境监测已经可以通过温湿度、光照和土壤水分等多类传感器实现连续采集。然而，完全依赖云端处理的传统架构在农业现场仍然存在明显局限：一方面，网络不稳定时自动控制可能失效；另一方面，图像等高维数据全部上传云端会增加带宽压力和隐私风险。对藏红花这类对环境极其敏感的高附加值作物而言，系统不仅需要“看得见”，还需要“在现场及时做出决策”。

因此，本文将边缘计算引入藏红花培育场景，在靠近数据源的边缘节点完成策略判断、数据存储和部分智能分析。该研究具有三方面意义：其一，通过本地闭环控制提升系统实时性与可用性；其二，通过边缘端处理视觉与控制任务，降低网络依赖和传输成本；其三，形成一套适合本科毕业设计落地的软硬件协同方案，为智慧农业和中药材培育的结合提供实践样本。

## 1.2 国内外研究现状

在智慧农业研究领域，基于物联网的环境监测已经较为成熟。温湿度、光照、土壤水分等参数的持续采集可显著提升种植管理的精细化程度，但许多系统仍采用“端—云”模式，现场设备主要承担上报功能，关键决策依赖云端完成。这种架构适用于网络条件较好的场景，但在温室、大棚或偏远种植点会受到网络质量制约。

边缘计算为农业控制提供了新的实现路径。树莓派等轻量级边缘节点兼具一定算力与完整操作系统能力，能够承担数据库、本地 Web 服务、规则推理和协议转换等任务。在农业灌溉、温室监测和设备联动等应用中，本地自治已被证明能够有效降低延迟并增强系统鲁棒性。

与此同时，机器视觉和人工智能也逐步应用到作物生长监测中。OpenCV 常用于颜色特征提取、图像阈值分割和目标状态识别，适合部署在边缘设备上；本地轻量化大模型则为农业问答与知识辅助带来了新的可能。不过，将多维环境感知、边缘自治控制、视觉识别和本地模型辅助同时结合到藏红花培育中的系统仍较少，这正是本文的主要切入点。

## 1.3 研究内容与创新点

围绕藏红花全生长周期培育需求，本文完成了以下研究内容：

1. 设计了以 STM32F411 为核心的底层感知与执行终端，实现温湿度、光照、土壤水分及手势信息的采集，并通过 USB 虚拟串口与树莓派进行双向通信。
2. 构建了以树莓派 4B 为核心的边缘服务端，基于 Flask 提供 Web 页面和 RESTful API，基于 SQLite 实现本地数据持久化与日志审计。
3. 设计了自动灌溉策略模块，使系统能够根据土壤湿度阈值、单次灌溉时长和冷却时间在边缘端自主决策。
4. 集成了 PAJ7620 手势识别模块与 OLED 菜单，实现农业现场的非接触式交互。
5. 在边缘端加入 OpenCV 视觉分析与 Qwen 量化模型推理，实现花期识别和自然语言种植建议。

相较于常见农业物联网系统，本文的创新点主要体现在四个方面：一是采用边缘自治架构，保证弱网环境下的持续运行；二是引入适合现场场景的非接触式交互方式；三是在端侧驱动中加入智能降级思路，提高传感器读取鲁棒性；四是尝试在树莓派有限算力下融合机器视觉与本地大模型，提升系统智能化水平。上述设计既回应了农业现场对可靠性的要求，也体现了毕业设计中软硬件协同实现的综合能力。

# 第二章 系统开发相关技术

## 2.1 边缘计算与物联网架构

物联网系统通常可划分为感知层、网络层和应用层。本文在此基础上结合实际需求，构建了“感知与执行层—边缘计算层—应用层”三层结构。感知与执行层负责采集环境参数并驱动水泵、灯带等执行器；边缘计算层承担数据接入、策略执行、数据存储和接口服务；应用层则面向用户提供监控、控制和分析界面。

边缘计算的核心价值在于将原本放在云端的部分能力前移到靠近数据源的位置。对于自动灌溉场景而言，边缘节点可直接读取本地传感器数据并立即做出响应，无需等待云端返回结果，从而提高实时性和可用性。对于图像分析场景，本地处理还可减少上传成本并增强隐私保护。与此同时，边缘端还承担协议转换、状态缓存和多线程调度等职责，是连接底层设备与上层应用的枢纽。

## 2.2 STM32 与 MicroPython

STM32F411 具备较好的主频、接口资源和扩展能力，能够满足多类传感器接入与执行控制需求。系统中 I2C 主要用于连接 BH1750、PAJ7620 和 OLED，ADC 用于采集土壤湿度模拟量，GPIO 用于 DHT11 通信和继电器控制。相较于较老的微控制器平台，STM32F411 在性能和可扩展性方面更适合承担多传感器并行采集任务。

在软件实现上，本文选用 MicroPython 作为端侧开发环境。其优势在于开发效率高、语法简洁、便于快速调试，并且与边缘端 Python 技术栈保持一致，有利于上下位机协同开发。借助面向对象封装，系统将各类传感器驱动抽象为独立模块，并在异常处理中加入分级降级机制，使底层程序在部分硬件读取失败时仍能维持主循环运行。对于本科项目而言，这种实现方式在开发效率和工程可控性之间取得了较好平衡。

## 2.3 Flask 与 SQLite

Flask 是轻量级 Web 框架，适合在树莓派等资源受限设备上部署。本文基于 Flask 实现实时数据查询、历史数据读取、设备控制、策略配置和身份认证等接口，并通过模板页面构建响应式 Web 监控界面。为保证控制功能安全，系统对高权限操作加入令牌鉴权机制，区分普通用户和管理员权限。

SQLite 作为嵌入式数据库，无需独立服务进程，部署与维护成本低，适合边缘侧单机持久化场景。结合 WAL 模式，SQLite 可以在高频写入与查询并存的条件下提升并发稳定性。系统围绕用户、角色、传感器数据、控制日志和灌溉策略等内容设计了多张业务表，满足历史追溯和系统管理需求。数据库的引入使系统从“只看实时值”进一步扩展到“可回顾、可分析、可审计”的阶段。

## 2.4 OpenCV 与本地大模型

OpenCV 在图像采集、颜色空间转换、阈值分割和特征提取方面具有良好的轻量化优势。本文在树莓派端使用摄像头采集藏红花图像，并将 RGB 图像转换为 HSV 空间，通过颜色阈值分割判断花色分布，从而辅助识别花蕾期、盛开期等生长阶段。

为了增强系统的人机交互能力，本文还在边缘端尝试部署量化后的 Qwen 模型。通过将视觉识别结果与实时环境数据拼接为提示词，模型能够输出简洁的种植建议。虽然边缘端算力有限，但轻量化模型已能够支撑较基础的农业问答和状态解释任务。与单纯显示数据相比，这种做法更贴近“辅助决策”而非“数据堆叠”的系统目标。

# 第三章 系统需求分析

## 3.1 业务痛点分析

传统藏红花培育流程中，种植人员需要多次进入种植场地观察温湿度、土壤水分和植株状态，并手动开启或关闭水泵、补光设备等执行器。该方式存在三个主要问题：一是数据获取不连续，难以及时发现突发环境变化；二是人工操作反应速度慢，容易导致调控滞后；三是管理依赖经验，缺乏记录与追溯机制，不利于规模化复制。

面向上述问题，系统需要实现从“人工巡检—经验控制”向“持续感知—本地决策—可视化管理”转变。也就是说，系统不仅要能采集数据，更要能围绕关键阈值自动触发动作，并将相关过程完整记录下来。此外，考虑到农业现场不一定具备稳定外网，系统必须保证最关键的灌溉和状态采集能力能够在局域网甚至离线场景下继续运行。

### 3.2.1 设备侧需求

设备侧需能够稳定采集温度、湿度、光照和土壤水分等环境参数，支持 OLED 本地显示，并能够控制水泵和 LED 灯带等执行器。此外，为适应农业现场手部沾泥、戴手套等场景，系统还需支持非接触式手势操作。

### 3.2.2 边缘服务端需求

边缘端需接收并解析来自 STM32 的串口数据，将实时数据存入数据库并向前端提供查询接口。同时，系统需提供历史数据可视化、用户登录、权限校验、控制日志记录和灌溉策略配置功能，以形成较完整的边缘网关能力。边缘端还要协调串口通信、页面访问和后台任务，避免单一模块阻塞影响整体系统。

### 3.2.3 自动控制需求

系统需根据土壤湿度等关键参数执行自动灌溉。策略设计应至少包含启停阈值、单次浇水时长和冷却时间，避免水泵在阈值附近频繁启停。控制过程应记录时间、动作和结果，便于后续审计与故障排查。

### 3.2.4 智能辅助需求

系统需支持使用摄像头采集作物图像，通过视觉算法识别花色和生长阶段，并结合实时环境数据生成基础建议。该功能不追求复杂大模型能力，而是强调在边缘设备上的可运行性和辅助决策价值。也就是说，智能模块应服务于种植管理，而不是脱离现场需求堆砌技术名词。

## 3.3 用户角色与非功能需求

系统主要面向两类用户：普通用户和管理员。普通用户可查看实时数据、历史曲线和部分状态信息；管理员在此基础上还可调整灌溉策略、执行远程控制并查看日志。这样的权限划分有助于避免误操作，提高系统安全性。

在非功能需求方面，系统首先要具备实时性，能够在较短时间内完成数据更新和控制响应；其次要具备高可用性，即在网络不佳甚至断网时保持本地核心功能正常；再次要具备安全性，重要操作需要进行身份校验；最后还要具备可维护性与可扩展性，便于后续增加更多传感器、执行器或智能分析模块。对于毕业设计而言，这些非功能需求同样重要，因为它们直接影响系统是否具有工程实现价值。


## 3.4 系统可行性与非功能需求

从可行性角度看，本文方案具备较好的技术可行性、经济可行性和实施可行性。技术上，STM32、树莓派、Flask、SQLite、OpenCV 及量化大模型均具有成熟开源生态，资料丰富、社区活跃，能够支撑系统从底层采集到上层分析的完整实现；经济上，所选硬件器件价格相对低廉，均属于高校实验室或个人开发可承受范围，适合作为毕业设计进行原型搭建；实施上，系统采用模块化设计，硬件接线、软件部署和参数调试均可分阶段推进，有利于降低开发风险，提高项目完成度。

在非功能需求方面，系统首先强调实时性。环境数据属于连续变化信息，若采集与展示间隔过长，将削弱监测价值，因此系统要求设备端持续稳定上报，边缘端快速解析并刷新页面，尽量将“采集—存储—展示”链路控制在较短延迟内。其次强调可靠性。农业现场容易受到潮湿、供电波动、传感器抖动及网络不稳定等因素影响，因此系统必须允许局部故障存在，不能因为某一路采集失败就使全局服务中断。为此，本文在底层驱动、串口通信和自动控制线程中都加入了异常捕获、状态回退和日志记录机制。

系统还强调安全性与可维护性。对于查看类功能，系统追求低门槛访问；对于策略修改、设备强制控制等高风险操作，则必须经过管理员身份校验并保留操作痕迹，确保后续能够追溯责任来源。与此同时，系统在代码结构上尽量保持松耦合：设备端以模块化驱动方式组织，边缘端将串口接入、数据持久化、业务逻辑和页面展示拆分为相对独立的部分。这样的设计不仅方便后续替换传感器、增加执行器，也为未来扩展短信告警、云端备份和移动端适配预留了空间。总体来看，上述可行性与非功能约束共同决定了系统不只是“能运行”，还应具备“能长期稳定运行”的工程属性。


# 第四章 系统总体设计

## 4.1 总体架构设计

系统整体架构分为设备端、边缘端和用户端三部分。设备端以 STM32F411 为核心，负责环境感知、状态显示和执行器驱动；边缘端以树莓派 4B 为核心，负责串口接入、数据库存储、Web 服务、自动灌溉和智能分析；用户端通过浏览器访问系统页面，查看数据并发送控制指令。三层之间形成“采集—处理—展示/控制”的闭环链路。

在通信上，STM32 与树莓派之间通过 USB 虚拟串口进行 JSON 数据交换。该方式同时兼顾供电、调试和数据传输，且实现难度较低。树莓派与用户端之间通过 HTTP 接口和 Web 页面进行交互，满足 PC 和移动端浏览器访问需求。通过这一架构，系统能够将实时控制和数据管理统一在本地边缘侧完成。

## 4.2 硬件选型与终端设计

核心主控选用 WeAct STM32F411CEU6，边缘节点选用树莓派 4B。传感器部分包括 DHT11 温湿度传感器、BH1750 光照传感器和电容式土壤湿度传感器；交互部分采用 PAJ7620 手势识别模块与 SSD1306 OLED；执行器部分包括水泵继电器和 LED 灯带。该方案器件常见、成本可控，适合毕业设计条件下完成搭建和调试。

终端电路设计重点在于接口匹配与供电稳定。I2C 总线用于连接多块数字器件，ADC 采集土壤湿度模拟量，继电器模块负责实现弱电控制强电负载。考虑到农业场景中的噪声和误操作风险，控制回路还需注意隔离与防抖，以保证执行动作可靠。整体方案遵循“先保证功能闭环，再逐步优化精度与工业化程度”的设计思路。

## 4.3 数据库设计

数据库围绕系统业务流程设计，核心表包括用户表、角色表、用户角色关联表、传感器数据表、控制日志表和灌溉策略表等。传感器数据表用于存储温湿度、光照、土壤湿度和时间戳；控制日志表用于记录水泵、灯带等执行器动作；灌溉策略表用于保存阈值、时长、启用状态等参数。通过这些表，系统能够同时支撑实时业务、历史分析和管理审计。

## 4.4 自动灌溉逻辑设计

自动灌溉采用规则驱动方式实现。系统周期性读取最新土壤湿度值，并将其与策略中的阈值比较。当湿度低于下限且未处于冷却期时，边缘端即下发开启水泵命令；当达到单次浇水时长后，再自动关闭水泵并进入冷却状态。这样既能保证及时补水，也能避免高频反复启停。

为提高可靠性，自动控制线程与手动控制共享串口资源时需要进行互斥管理，防止同时下发命令造成状态混乱。此外，所有控制动作都写入日志，以便追踪自动和手动操作的来源与结果。该逻辑虽然基于规则，但实现简单、解释性强，适合作为系统初版的控制核心。

## 4.5 容错与安全设计

为了保证系统在真实农业场景中的可持续运行，本文在总体设计阶段同步考虑了容错与安全问题。首先，在容错方面，设备端与边缘端都采用“关键链路优先、局部异常隔离”的原则。设备端主循环不会因为单个传感器短时失效而中断，而是通过重试、降级或保留上一时刻有效值来维持数据流连续性；边缘端在串口接收时对异常 JSON、空包和超时情况进行过滤，避免脏数据直接进入数据库或触发错误控制。其次，在安全方面，系统将查看权限与控制权限分离，普通浏览请求可以快速获取状态信息，而涉及灌溉策略、设备启停等动作的接口则必须通过令牌鉴权。对于每一次控制行为，系统都会记录触发来源、执行时间和设备对象，为后续排查误操作、异常灌溉和设备故障提供依据。该设计虽然仍属于轻量级实现，但已经体现出完整的工程防护思路。


# 第五章 系统功能实现

## 5.1 端侧数据采集与交互实现

在 STM32 端，系统将各类传感器驱动封装为独立模块，由主循环按固定周期采集数据并打包为 JSON 结构发送给树莓派。为了避免阻塞影响整体响应，主程序采用非阻塞循环方式，在不同时间片中分别完成数据采集、OLED 刷新、手势识别和串口命令接收。

针对传感器现场不稳定问题，系统加入了降级机制。以温湿度采集为例，当硬件读取失败达到一定次数后，程序可退回到备选读取路径，必要时再进入兜底模式，从而避免单个模块异常拖垮整个主循环。这一实现方式提升了系统在实验环境中的连续运行能力，也体现了嵌入式设计中“容错优先”的思路。

PAJ7620 手势识别模块与 OLED 菜单结合后，能够完成页面切换、选项移动和控制触发等操作。通过加入冷却时间，可减少一次挥手被重复识别的问题，提高交互准确率和现场使用体验。相比直接增加更多按键，手势交互更契合农业现场的清洁性和便捷性需求。

## 5.2 边缘服务与前后端实现

树莓派端运行 Flask 服务，负责启动串口读取线程和自动灌溉线程，并向前端提供实时数据、历史数据、控制接口和策略配置接口。前端页面包含实时监控页、历史数据页和后台管理页。实时页面展示温度、湿度、光照、土壤湿度及手势状态；历史页面以折线图方式展示环境变化趋势；后台管理页用于调整自动灌溉参数和查看控制日志。

## 5.3 自动灌溉与边缘智能实现

自动灌溉模块由独立守护线程持续运行。线程定期读取最近一条土壤湿度数据和当前策略参数，满足条件时向设备端发送开泵命令，定时结束后再发送关泵命令，并写入两条对应日志。通过冷却机制，系统可有效避免因湿度波动导致的频繁灌溉。管理员还可通过后台页面修改阈值、启用状态和浇水时长，使控制逻辑具备一定的现场适配能力。

在智能分析方面，系统通过摄像头拍摄藏红花图像，调用 OpenCV 进行 HSV 分析，根据颜色区域比例判断花期状态。随后再将花期结果与温湿度等环境参数整合为提示词，交由本地量化后的 Qwen 模型生成建议。这样一来，系统不仅能告诉用户“当前数值是多少”，还能够进一步解释“这些数值意味着什么以及应该如何处理”，从而提升人机交互层次。

## 5.4 数据流闭环与关键流程实现

从系统运行流程看，整个平台形成了较完整的数据闭环。首先，STM32 周期性采集环境参数，并通过 USB 虚拟串口将 JSON 数据发送到树莓派；其次，树莓派中的串口线程完成解析，将最新值刷新到内存，同时写入 SQLite 形成历史记录；然后，前端页面通过异步请求读取实时数据与历史数据，用户可根据页面状态进行观察或下发控制指令；最后，自动灌溉线程和人工控制接口又会将指令经串口发送回设备端，由继电器驱动水泵或灯带完成动作。也就是说，系统不是孤立的“采集系统”或“网页系统”，而是实现了“数据进入—边缘处理—用户决策/自动决策—设备执行—结果再反馈”的完整闭环。

在这一闭环中，互斥与异步机制尤为关键。一方面，串口既承担数据上报，又承担控制命令下发，若缺乏锁机制，自动灌溉与人工控制可能同时写入串口缓冲区，导致命令混乱；另一方面，页面查询与数据库写入是高频并发行为，若全部在单线程中串行执行，将直接影响实时显示效果。因此，本文通过线程分工、共享状态缓存和数据库 WAL 模式，兼顾了响应速度与运行稳定性。该实现说明，在资源有限的边缘节点上，只要设计得当，仍然可以支撑较完整的监测与控制业务流程。

## 6.4 测试结果讨论

综合各项测试可以看出，本文系统较好地达成了毕业设计的预期目标。首先，在功能完整性方面，系统已经实现了从底层采集、边缘接入、前端展示到自动控制与智能分析的全链路闭环，这意味着其不再停留在“单点功能演示”层面，而具备了较完整的应用雏形。其次，在工程可用性方面，断网自治、日志审计和错误降级等机制的加入，使系统具备一定的现场运行能力，说明边缘计算架构确实能够提升农业场景中的可靠性。

当然，测试结果也暴露出一些值得继续优化的问题。其一，当前视觉识别仍以颜色阈值法为主，易受到光照条件和拍摄角度影响，后续可增加图像预处理、样本数据集和轻量级分类模型，以提高识别准确率。其二，本地大模型虽然能生成较自然的建议，但推理时间仍偏长，且建议质量会受提示词组织方式影响，说明模型模块目前更适合作为辅助工具，而非唯一决策依据。其三，自动灌溉策略主要依赖固定阈值，缺少对天气、作物阶段和历史趋势的综合考虑。总体而言，这些问题并不否定系统设计本身，而是表明该原型已经具备继续深化为更高质量系统的基础。此外，从论文写作角度看，本系统的价值不仅体现在功能实现上，还体现在其验证了边缘计算在小型专用农业场景中的适配性：当算力、成本和部署条件都受限时，采用“轻量感知 + 本地规则 + 按需智能”的组合，往往比一味追求复杂模型或全云架构更符合实际应用需求。这一结论对类似中药材、花卉或小型温室作物的数字化改造也具有一定参考意义。


# 第七章 总结与展望

## 7.1 全文总结

本文围绕藏红花全生长周期培育需求，设计并实现了一套基于边缘计算的智能培育系统。系统以 STM32F411 完成多源环境感知与执行控制，以树莓派 4B 完成边缘接入、数据库存储、Web 服务和自动灌溉决策，并通过 OpenCV 与本地量化大模型扩展了视觉识别和专家建议功能。测试结果表明，该系统能够较稳定地完成环境监测、历史记录、远程控制、手势交互和自动灌溉等任务，在弱网或断网条件下仍具备本地闭环控制能力。

从工程实践角度看，本文验证了边缘计算在高附加值作物培育中的应用价值，也说明了将嵌入式控制、轻量级边缘服务与基础 AI 能力结合到同一平台是可行的。对于本科毕业设计而言，该系统同时覆盖了硬件设计、通信协议、数据库、Web 开发、自动控制和智能分析等多个技术点，具有较好的综合性和展示价值。更重要的是，本文并未将这些技术简单拼接，而是围绕藏红花培育这一明确场景组织成具有内在逻辑的完整系统：监测服务于控制，控制服务于培育，智能分析服务于理解和决策。这种以应用目标牵引技术实现的思路，也体现了系统设计的整体性。

## 7.2 不足与展望

当前系统仍存在一些不足。首先，DHT11 等低成本传感器更适合原型验证，在精度、耐久性和抗干扰性方面仍有提升空间；其次，树莓派的边缘算力有限，复杂视觉任务和更大参数量模型难以流畅运行；再次，自动灌溉策略主要基于固定阈值，尚未结合更多时序特征和预测能力；最后，系统目前偏向单节点实验场景，在多节点协同、云边协同、移动端接入和远程运维方面仍有扩展空间。

后续研究可从四个方向展开：一是升级传感器与执行器，提高测量精度与长期稳定性；二是引入带 NPU 的边缘设备或采用云边协同架构，提升智能分析能力；三是结合历史数据建立更动态的灌溉模型，实现更精细的调控；四是扩展多终端接入和多节点管理能力，推动系统从实验验证走向更真实的种植应用。若后续能够结合更规范的传感器标定、更多季节数据以及实际种植反馈，系统的实用价值还可进一步提升。

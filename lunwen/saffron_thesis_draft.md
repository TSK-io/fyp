# 摘要

藏红花属于高附加值经济作物，其生长周期长、环境敏感性强，对温湿度、光照、土壤含水率以及人工巡检频率均提出了较高要求。传统家庭式或小规模培育方式往往依赖人工经验完成浇水、补光、观察花色与记录数据，不仅劳动强度较大，而且存在数据离散、决策滞后、设备联动能力不足等问题。针对上述问题，本文结合边缘计算、嵌入式控制、Web 可视化和轻量化人工智能技术，设计并实现了一套基于边缘计算的藏红花全生长周期智能培育系统。

本系统采用“设备端—边缘端—用户端”三层架构。设备端以 STM32F411 微控制器为核心，基于 MicroPython 实现 DHT11 温湿度采集、BH1750 光照采集、ADC 土壤湿度采集、PAJ7620 手势识别、本地 OLED 显示以及水泵、灯带等执行器控制；边缘端基于 Raspberry Pi 4B 部署 Flask 服务、SQLite 数据库、串口采集线程和自动灌溉线程，完成实时数据接收、历史数据持久化、策略控制、用户鉴权、控制日志记录和 CSV 导出；用户端通过浏览器访问实时监控界面、历史趋势界面和后台管理界面，实现环境查询、远程控制、策略配置和运行状态查看。系统进一步引入 OpenCV 颜色分析方法对藏红花图像进行生长阶段辅助判定，并预留本地轻量化大模型接口，用于给出简要种植建议，体现了边缘智能在农业场景中的应用价值。

从代码实现结果看，系统已形成较完整的软硬件闭环：STM32 周期性上报 JSON 数据，树莓派串口线程解析后写入数据库，同时自动灌溉线程依据土壤阈值控制水泵启停；前端页面支持实时刷新、历史曲线展示、控制日志查询、登录鉴权和策略配置。本文围绕总体方案、器件选型、硬件设计、软件设计、数据库设计与测试方案进行了系统阐述，并给出适用于毕业论文排版的结构化 Markdown 文稿。实践表明，该方案具有低成本、可扩展、部署灵活和功能集成度较高等特点，能够满足中小规模藏红花智能培育场景下的数据采集、边缘处理与可视化管理需求。

**关键词：** 边缘计算；藏红花培育；STM32F411；树莓派；Flask；物联网；智能灌溉；机器视觉

# Abstract

Saffron is a high-value crop with a long growth cycle and strong sensitivity to environmental conditions. Traditional cultivation methods rely heavily on manual observation and empirical management, which often lead to delayed decision-making, insufficient data accumulation, and weak device coordination. To address these issues, this thesis designs and implements an intelligent saffron cultivation system based on edge computing, integrating embedded control, Web services, database management, computer vision, and lightweight AI techniques.

The system adopts a three-layer architecture consisting of the device layer, edge layer, and user layer. The device layer is built on an STM32F411 microcontroller running MicroPython, responsible for temperature and humidity acquisition, light sensing, soil moisture collection, gesture interaction, local OLED display, and actuator control. The edge layer is deployed on a Raspberry Pi 4B, where Flask, SQLite, serial communication threads, and automatic irrigation threads are used to support real-time data reception, historical storage, control policy execution, authentication, logging, and CSV export. The user layer provides browser-based interfaces for real-time monitoring, historical trend viewing, and background administration. In addition, OpenCV-based color analysis is introduced for auxiliary growth-stage recognition, and a lightweight local language model interface is reserved for cultivation suggestions, demonstrating the practical value of edge intelligence in agricultural scenarios.

The implementation shows that the proposed system forms a complete hardware-software loop, including periodic JSON data transmission from STM32, edge-side parsing and storage on Raspberry Pi, automatic irrigation based on threshold strategies, and browser-side visualization and control. The system is characterized by low cost, flexible deployment, and strong extensibility, making it suitable for small- and medium-scale saffron cultivation scenarios.

**Key Words:** edge computing; saffron cultivation; STM32F411; Raspberry Pi; Flask; Internet of Things; intelligent irrigation; computer vision

# 1 绪论

## 1.1 研究背景

随着物联网、边缘计算和智能农业技术的快速发展，传统农业生产方式正逐步由经验驱动向数据驱动转变。尤其在花卉、中药材及高经济价值作物培育领域，环境控制精细度直接影响产量、品质和人工成本。藏红花作为一种药食两用且经济价值较高的特色作物，其花期、温湿度适应区间、土壤水分条件及光照环境均具有较高敏感性。如果培育过程中缺乏连续监测与及时调控，容易出现土壤过湿导致球茎腐烂、光照不足导致长势不良、巡检不及时导致花色与生长阶段判断滞后等问题。

在传统培育模式下，管理者通常通过温湿度计、人工浇水、经验判断和纸质记录来完成日常管理。这种方式虽然成本较低，但存在四个明显不足：一是环境数据采样频率低，难以形成连续的历史趋势；二是人工控制存在主观性，浇水和补光时机不稳定；三是远程访问能力弱，无法随时查看培育状态；四是系统通常缺乏图像分析和智能建议功能，不利于后续扩展。因此，设计一套面向藏红花培育场景的低成本、可联网、可视化、具备边缘处理能力的智能系统，具有较强的现实意义和应用价值。

本课题的研究建立在现有开源软硬件生态基础之上：底层采用 STM32F411 单片机承担多传感器采集与本地控制，中间层采用树莓派执行数据汇聚、策略计算与 Web 服务，上层通过浏览器实现跨终端访问。该方案兼顾了嵌入式实时性、边缘计算灵活性和应用界面友好性，适合本科毕业设计在成本、复杂度和展示效果之间取得平衡。

## 1.2 研究目的及意义

本课题的直接目标是完成一套能够服务于藏红花培育过程的智能监测与控制系统，并围绕系统设计、开发与验证形成完整的毕业论文成果。具体而言，研究目的主要体现在以下几个方面。

首先，通过多传感器融合采集实现温度、湿度、光照和土壤湿度的连续监测，使培育环境由“凭经验判断”转为“以数据为依据”。其次，通过设计自动灌溉策略和远程控制接口，使系统具备基础的环境干预能力，从而减轻人工操作负担。再次，通过 Web 前端、数据库和日志机制实现实时显示、历史追溯与权限管理，提升系统的工程完整度。最后，通过引入机器视觉和轻量化本地模型接口，为系统增加“看得见”和“会分析”的能力，使其不再停留于单纯的传感器采集平台，而是向边缘智能系统演进。

本课题的研究意义主要包括理论意义和实践意义两个层面。在理论层面，本文通过将边缘计算架构、嵌入式设备协同、轻量化数据库设计、前后端分离思想以及简化的视觉分析方法融合到一个具体农业场景中，形成一套可解释、可实现、可扩展的系统方案，为同类中小型智能农业系统设计提供参考。在实践层面，本系统使用的 STM32F411、树莓派、DHT11、BH1750、土壤湿度传感器、继电器和 USB 摄像头等器件均具有成本可控、采购方便和开发生态成熟等特点，适用于原型验证和教学实践，具有较高的落地价值。

## 1.3 国内外研究现状

### 1.3.1 国内研究现状

国内智能农业研究近年来持续升温，温室监测、智能灌溉、作物识别和农业物联网平台成为高校与企业的重点方向。多数研究聚焦于环境参数采集、无线通信、云平台监控和自动控制等功能，一些项目已能够实现基于土壤湿度的自动浇灌、基于手机 App 的远程管理以及基于摄像头的病虫害识别。对于本科毕业设计而言，常见实现路线包括以 STM32、Arduino 或 ESP32 为核心的终端采集方案，以及以 Flask、Django 或微信小程序为上层展示平台的轻量化应用方案。

然而，从实际作品质量看，许多系统仍存在功能割裂的问题。例如，硬件端仅实现简单采集而缺少完整数据库设计；Web 端仅展示静态页面而未与真实设备形成闭环；控制逻辑可用但缺乏日志审计和角色权限；部分“智能”模块只是概念性堆砌，没有与业务数据真正联动。就藏红花这一相对细分作物而言，专门面向其全生长周期的边缘智能培育系统公开案例并不多见，多数研究仍停留在一般花卉或温室作物通用监测层面。

### 1.3.2 国外研究现状

国外在精准农业、边缘人工智能和农业机器人方面起步较早，相关研究更强调系统稳定性、模型泛化能力和规模化部署。许多研究采用 LoRa、ZigBee、Wi-Fi、MQTT 等通信机制构建分布式监测网络，并将边缘节点用于数据预处理、事件检测和推理加速，以降低云端负担和通信延迟。在作物表型分析方面，计算机视觉已被广泛用于花朵识别、生长状态评估、成熟度判断和病害检测。

从技术演进趋势看，国外研究逐渐呈现三方面特点：一是强调边缘计算与云计算协同，通过边缘节点处理实时任务、云端负责长期建模；二是强调多模态融合，将传感器数据与图像数据结合起来进行综合判断；三是强调平台化和标准化，采用 REST API、消息队列和标准化数据库接口提升系统复用性。相比之下，本课题虽然规模较小，但在结构上已体现出多模态感知、边缘决策、权限管理和可视化展示等工程要素，具备继续向更高层次扩展的基础。

## 1.4 研究主要内容

围绕系统目标，本文主要完成以下研究内容。

第一，完成系统需求分析与总体方案设计，明确系统需要满足的实时采集、远程监控、自动灌溉、身份鉴权、历史存储和可视化分析等功能，并提出分层式总体架构。第二，完成设备端硬件与嵌入式程序设计，以 STM32F411 为核心实现多传感器采集、OLED 页面显示、手势交互及执行器控制。第三，完成边缘端软件系统设计，以树莓派为载体搭建 Flask 服务、SQLite 数据库、串口采集线程和自动灌溉线程，实现数据持久化与业务逻辑闭环。第四，完成前端页面与接口设计，实现实时监控、历史曲线、后台管理、登录注册和策略配置。第五，在系统中加入图像采集、颜色识别和种植建议接口，形成基础的边缘智能功能。第六，设计测试方案并分析结果，评价系统的功能完整性、稳定性与扩展潜力。

# 2 相关理论与技术基础

## 2.1 边缘计算与农业物联网基础

边缘计算是指将部分计算、存储和决策能力从中心云下沉到靠近数据源的网络边缘节点，从而降低传输延迟、减轻网络压力并提高系统实时性。对于智能农业场景而言，传感器采集频率高、控制反馈要求及时、网络环境可能不稳定，如果所有数据都依赖远程云端处理，容易出现响应慢、可靠性低和部署复杂的问题。将边缘节点部署在树莓派等本地计算设备上，可以在靠近作物的地方完成数据解析、短期存储、阈值判断、图像分析和本地页面服务，更适合中小规模实验环境。

农业物联网通常包括感知层、网络层和应用层。感知层负责温湿度、光照、土壤、水位、图像等信息获取；网络层负责 USB、串口、Wi-Fi、局域网和未来可扩展的 MQTT 等通信机制；应用层负责可视化、报表、控制策略和用户交互。本系统在此基础上进一步强化边缘层角色，将树莓派视为设备侧与用户侧之间的中枢，实现了“采集—处理—存储—控制—展示”的完整数据链路。

## 2.2 嵌入式多传感器采集技术

嵌入式系统是智能农业终端的核心组成部分，其任务包括采集传感器数据、驱动显示与执行器、完成本地交互以及与上位机进行数据通信。本系统选用 STM32F411 作为主控芯片，并在其上运行 MicroPython 固件，以降低开发门槛、加快调试效率。相比传统裸机 C 语言方式，MicroPython 在原型系统开发中更适合快速验证功能逻辑，尤其有利于本科阶段对软硬件协同流程的整体掌握。

在多传感器采集方面，本系统使用 DHT11 测量温湿度，BH1750 测量光照，ADC 模拟输入读取土壤湿度传感器，PAJ7620 实现手势识别，OLED 用于显示主要状态。由于不同器件接口类型不同，系统需要协调 GPIO、I2C 与 ADC 等多种外设资源。为提高可靠性，温湿度驱动采用了硬件级、软件级和模拟级三重降级策略，即当高优先级驱动不可用时自动切换到次一级驱动，以保证系统在传感器异常或底层支持不足时仍可持续运行。

## 2.3 串口通信与 JSON 数据交换技术

在本系统中，STM32 与树莓派之间通过 USB 虚拟串口连接完成供电与数据通信。串口通信的优势在于实现简单、连接稳定、调试方便，特别适合作为毕业设计原型中的设备间桥梁。为了兼顾可读性与扩展性，系统采用 JSON 文本作为数据交换格式。STM32 端周期性输出包含温度、湿度、光照、土壤湿度、手势信息和循环计数的 JSON 数据包，树莓派端读取后再解析写库；控制命令同样采用 JSON 形式，如 `{\"actuator\":\"pump\",\"action\":\"on\"}`，可直接扩展到更多执行器。

这种设计虽然相较于二进制协议在传输效率上略低，但对于本系统的数据规模和实时要求而言完全可接受，且具备结构清晰、调试友好和易于跨语言解析等优点，非常适合原型系统快速迭代。

## 2.4 Web 服务、SQLite 与角色权限控制技术

边缘端软件部分基于 Flask 框架构建 REST 风格接口与页面渲染服务。Flask 具有轻量、灵活、学习成本低等特点，适用于树莓派环境部署。本系统通过 Flask 实现了实时数据获取接口、历史数据查询接口、控制命令接口、灌溉策略接口、登录注册接口和设备状态接口，并通过 HTML、CSS 与 JavaScript 构建了实时监控页面、历史曲线页面、后台管理页面和登录页面。

数据库采用 SQLite。考虑到本系统初期部署在树莓派本地，SQLite 具备零配置、占用资源小、便于移植等优势，适合本课题原型阶段使用。系统共设计用户表、角色表、用户角色关联表、设备表、传感器数据表、控制日志表和灌溉策略表等多张业务数据表，并通过触发器实现控制日志写入后自动更新设备在线时间。针对权限控制，系统采用用户名密码注册登录、密码哈希存储、基于令牌的身份验证以及管理员角色校验等方式，提升了后台操作安全性和系统工程完整度。

## 2.5 机器视觉与轻量化边缘智能技术

在传统环境参数采集基础上，仅依靠温湿度和土壤数据难以直接判断花朵颜色变化和阶段特征。因此，本系统引入 USB 摄像头与 OpenCV 图像处理模块，通过 HSV 颜色空间阈值分割方法对红色、绿色和粉色区域进行统计，并将结果映射到“花蕾期”“盛开期”“成熟或凋谢期”等状态，实现对藏红花生长阶段的辅助识别。该方法计算量较小，适合在树莓派边缘端本地运行。

此外，系统还预留了轻量化本地大模型接口。其核心思想不是替代农业专家，而是根据当前环境参数生成简短的种植建议，如是否需要浇水、是否光照偏弱等。这种“传感器数据 + 边缘推理 + 自然语言反馈”的方式，为后续实现更复杂的农业知识问答和规则辅助决策奠定了基础。

# 3 系统总体方案设计

## 3.1 系统需求及可行性分析

### 3.1.1 需求分析

结合课题目标与代码实现情况，系统应至少满足以下需求。

一是环境感知需求。系统应能够稳定采集藏红花培育环境中的温度、湿度、光照和土壤湿度，并具备一定刷新频率，支持实时查看与历史回溯。二是本地交互需求。终端设备应具备基础的人机交互能力，能够在 OLED 屏上显示当前数据，并通过手势切换页面、选择控制项，增强现场演示效果。三是远程监控需求。用户应通过手机或电脑浏览器访问实时页面与历史页面，查看曲线、时间戳与状态。四是控制需求。系统应支持手动控制水泵、状态灯和灯带，并在满足条件时执行自动灌溉。五是管理需求。系统应区分普通查看与管理员操作，通过登录、角色与日志机制保证关键操作可追溯。六是扩展需求。系统应预留摄像头、视觉分析和云端通信能力，以便后续升级。

### 3.1.2 可行性分析

从技术可行性看，系统所使用的 STM32F411、树莓派、Flask、SQLite、OpenCV 和 MicroPython 等技术栈成熟度较高，相关文档丰富，开发和调试成本适中。设备端传感器和执行器接口类型清晰，树莓派具备足够的计算资源承载 Web 服务、串口监听、图像处理和轻量推理。软件上采用 Python 生态也有利于快速开发。

从经济可行性看，本系统主要使用低成本教学实验硬件与开源软件，不需要额外采购昂贵工业设备。对于本科毕业设计而言，这种配置可以在有限预算内实现较为丰富的功能展示。

从实施可行性看，系统分层明确，可先完成基础采集和显示，再逐步叠加数据库、页面、控制、视觉和智能建议模块，便于阶段性验收。即使个别高级功能未完全达到最终目标，核心监测与控制链路仍能独立工作，说明系统具备较强的可迭代性。

## 3.2 系统总体方案设计

本系统总体采用三层架构设计。

第一层为设备端，主要由 STM32F411、DHT11、BH1750、土壤湿度传感器、PAJ7620 手势模块、OLED 显示屏、水泵继电器和 LED 灯带继电器组成。该层负责环境数据采集、本地显示、手势控制以及接收边缘端下发的控制命令。

第二层为边缘端，采用 Raspberry Pi 4B 作为边缘计算节点。该层通过 USB 虚拟串口与 STM32 相连，运行 Flask 服务、串口读取线程、自动灌溉线程、SQLite 数据库和图像处理模块。边缘端既是数据汇聚中心，也是控制逻辑执行中心和人机交互服务中心。

第三层为用户端，用户通过浏览器访问系统页面，包括实时监控、历史曲线、后台管理和登录界面。未来可在此基础上扩展至手机 App、微信小程序或云服务器。

系统数据流如下：STM32 周期性采集传感器数据并打包成 JSON，经串口发送至树莓派；树莓派解析后更新实时缓存并写入数据库；Web 接口从缓存与数据库中提供实时数据和历史记录；管理员可通过页面设置灌溉策略或手动发送控制命令；自动灌溉线程根据最新土壤湿度和策略参数控制水泵启停，并将操作结果写入控制日志。

## 3.3 主要功能模块的器件选型

### 3.3.1 主控芯片选型

本系统设备端主控选择 STM32F411。相较于 51 单片机，STM32F411 具备更高主频、更丰富外设资源和更好的扩展能力；相较于 ESP8266/ESP32 等 Wi-Fi 芯片，STM32F411 更适合作为稳定的底层采集控制核心，并可通过树莓派承担上层网络与计算任务。结合课题对多传感器接入、I2C 扩展、ADC 采样和串口通信的需求，STM32F411 能够较好满足要求。

### 3.3.2 边缘节点选型

边缘节点选用 Raspberry Pi 4B。与普通单片机相比，树莓派运行 Linux 系统，具备更完整的软件生态，适合部署 Python、Flask、OpenCV、数据库和轻量模型。与直接将所有功能放在云服务器相比，树莓派部署在本地更有利于串口接入、低延迟控制和离线运行。

### 3.3.3 传感器与执行器选型

温湿度模块选用 DHT11，成本低、资料丰富、适合原型验证；光照模块选用 BH1750，接口简单、输出为数字量，便于快速接入；土壤湿度模块采用模拟输出方式，通过 STM32 ADC 采样并进行百分比换算；交互模块选用 PAJ7620 手势传感器，用于提升现场演示性和本地交互性；显示模块选用 0.96 英寸 OLED，功耗低、显示清晰；执行部分采用继电器控制水泵和 LED 灯带，能够直观体现系统调控效果。

### 3.3.4 数据库与软件框架选型

数据库选用 SQLite，原因在于其无需独立服务、适合边缘设备部署、支持事务和基本 SQL 功能，能够满足本课题原型阶段需求。后端框架选用 Flask，前端采用原生 HTML、CSS、JavaScript 以及 ECharts 绘图组件，实现速度快，且便于在毕业设计中展示接口与页面逻辑。

# 4 系统硬件设计

## 4.1 系统整体硬件工作原理

系统硬件由感知单元、控制单元、显示单元和执行单元组成。DHT11 负责采集空气温湿度，BH1750 负责采集环境光照强度，土壤湿度传感器输出模拟电压供 STM32 ADC 采样，PAJ7620 通过 I2C 接口采集手势信息，OLED 显示模块实时展示监测值与控制状态。STM32 周期性读取各类数据，按照预设逻辑完成显示更新，并通过 USB 虚拟串口向树莓派上传 JSON 数据。当树莓派根据管理员操作或自动灌溉策略发出控制命令后，STM32 接收命令并切换继电器状态，从而控制水泵与灯带。

与单纯的传感器采集系统相比，本系统硬件链路更强调“采集—显示—通信—执行”的闭环。设备端既是感知节点，也是局部控制节点；边缘端不是简单转发器，而是具备策略判断与多业务整合能力的计算节点。

## 4.2 系统功能模块设计

### 4.2.1 温湿度采集模块

温湿度采集模块使用 DHT11 传感器，通过单总线方式与 STM32 相连。代码中将其封装为 `DHT11Sensor` 类，并采用三级驱动降级机制：优先尝试硬件级读取，其次尝试软件驱动，若连续失败则启用模拟模式。该设计的优点在于，在课程设计环境下即使硬件接触不良或固件支持有限，系统仍可保持整体逻辑运行，降低演示失败风险。

### 4.2.2 光照采集模块

光照采集模块使用 BH1750，通过 I2C 总线连接。程序中对其进行了简化封装，初始化阶段发送连续高分辨率模式命令，并在读取时将原始数据转换为 lux 值。由于 OLED 和 PAJ7620 也挂接在同一 I2C 总线上，因此系统通过总线共享的方式节省了引脚资源，提高了接口利用率。

### 4.2.3 土壤湿度采集模块

土壤湿度模块采用模拟量输出方式，通过 STM32 的 ADC 引脚采样。程序根据经验值设置了干燥阈值 `DRY=59000` 与湿润阈值 `WET=26000`，并通过线性归一化公式将采样值换算为百分比。该方法实现简单、直观易懂，能够满足原型系统中土壤干湿状态的近似表征需求。在正式答辩前，建议结合实际传感器标定结果对阈值进行微调，以获得更准确的百分比表示。

### 4.2.4 手势识别与本地显示模块

手势识别模块采用 PAJ7620，通过 I2C 接口接入。系统将其作为本地交互的重要组成部分：在主页面可显示当前手势名称，在控制页面可通过左右切页、前后选择、上下执行等方式控制执行器状态。该设计增强了系统的人机交互体验，使设备端不仅是“数据终端”，也是“交互终端”。

本地显示模块采用 128×64 OLED 屏幕，设计为主页面、控制页面和信息页面三种显示模式。主页面展示温湿度、光照、土壤湿度和手势信息；控制页面展示水泵、灯带和状态指示灯状态；信息页面显示驱动模式、循环计数和 MicroPython 版本。多页面设计提高了信息组织性，适合现场演示。

### 4.2.5 执行器与供电模块

执行器部分主要包括水泵和 LED 灯带。程序通过继电器输出控制其通断，水泵用于模拟灌溉，灯带可视为补光或状态展示装置。状态指示灯则用于系统运行指示。由于水泵等负载与控制板电流需求不同，在实际接线时应注意电源隔离、地线共地以及继电器模块驱动能力，避免因瞬态电流造成控制异常。

若后续制作 PCB，可将 STM32 核心板、电源接口、传感器接口、继电器驱动接口和调试接口统一规划，以提高系统稳定性和布线整洁度；若采用面包板原型，也应尽量缩短高频率通信线路并固定模块位置，降低接触不良概率。

## 4.3 系统 PCB 设计

当前项目更偏向功能原型验证，硬件实现可基于面包板或底板方式完成。就毕业设计文稿而言，仍有必要对 PCB 设计思路进行说明。PCB 设计应遵循模块化原则，将核心控制区、传感器接口区、电源区和继电器驱动区分开布置，减少干扰与排线复杂度。I2C 总线应尽量缩短并预留上拉，ADC 模拟采样线应远离继电器驱动回路，水泵与灯带等功率负载区域应考虑隔离与保护。若后续进行正式制板，可在论文中补充原理图、PCB 走线图与实物图，以提升硬件完成度。

# 5 系统软件设计

## 5.1 系统软件整体设计

软件系统由设备端固件、边缘端服务程序、数据库模块和前端页面四部分组成。设备端程序承担实时采集、本地显示和命令执行；边缘端负责串口监听、业务逻辑调度、图像处理与接口服务；数据库负责历史存储与权限数据管理；前端页面承担人机交互与数据展示。四部分之间通过串口 JSON、HTTP 请求和 SQL 操作相互联动，形成完整的软件体系。

从结构上看，本系统并非简单的单脚本堆叠，而是具有一定模块化特征。设备端将 DHT11、PAJ7620、OLED 等驱动拆分到独立库文件中；边缘端将数据库操作抽离为 `db.py`，将页面拆分为多个模板，将控制逻辑划分为串口读取、自动灌溉、视觉分析和鉴权等子模块。这种结构有利于后期维护与扩展。

## 5.2 设备端软件设计

设备端主程序运行于 STM32F411 的 MicroPython 环境中。程序启动后首先完成各类硬件初始化，包括 DHT11、I2C 总线、BH1750、OLED、PAJ7620、土壤湿度 ADC、水泵继电器和灯带继电器。若初始化成功，OLED 将显示系统启动信息；若个别模块异常，则程序记录错误但尽量保证其他模块继续运行，以体现较好的容错思路。

在主循环设计上，程序采用非阻塞方式运行。一方面，通过 `select.poll()` 检测标准输入，实现串口命令的异步处理；另一方面，通过时间片判断每秒执行一次传感器采样与数据上报。同时，手势处理逻辑以较高频率轮询，支持页面切换和控制操作。这种写法避免了在单线程环境中因阻塞式等待导致采样、交互和通信互相影响的问题。

设备端命令处理逻辑较为清晰：对于结构化 JSON 命令，解析执行器名称与动作字段，控制水泵和灯带；对于简单字符串命令，则控制状态 LED 开关。程序执行后会打印反馈 JSON，便于树莓派或串口终端确认执行结果。

设备端每秒构造一次数据包，字段包括循环计数、时间戳、当前手势、温度、湿度、光照和土壤湿度。数据包通过 `print(json.dumps(...))` 输出到串口，由树莓派端统一接收。该做法实现简单，调试时可直接在串口工具中查看输出内容。

### 5.2.1 设备端主循环流程

设备端程序运行流程可概括为：系统初始化 → 手势轮询 → 串口命令处理 → 周期性传感器采集 → JSON 数据上报 → OLED 刷新显示 → 进入下一轮循环。若手势有效，则更新页面或触发控制；若接收到控制命令，则立即驱动继电器状态变化；若达到采样时间，则读取各类传感器并刷新缓存。

### 5.2.2 传感器驱动设计

设备端驱动模块中，DHT11 驱动具有较强的代表性。其实现不仅封装了测量与取值接口，还维护了驱动模式、错误统计和降级策略。该模块体现了面向对象封装思想，有利于代码复用和稳定性提升。类似地，BH1750 与 PAJ7620 模块也分别封装为独立类或独立驱动文件，便于在主程序中统一调用。

## 5.3 边缘端软件设计

边缘端软件部署在树莓派上，以 Flask 为核心构建统一服务。系统启动后，首先初始化数据库、默认设备信息、摄像头目录和全局缓存，然后开启串口读取线程与自动灌溉线程，最后监听 5000 端口提供页面和接口服务。整体上属于“单服务进程 + 多后台线程 + 数据库持久化”的轻量级架构。

### 5.3.1 串口读取线程设计

串口读取线程固定监听 `/dev/ttyACM0`，波特率为 115200。线程持续读取串口数据，当检测到包含温湿度字段的 JSON 文本后，即完成解码、解析、时间戳附加、最新数据缓存更新以及历史数据写库。该线程同时负责更新设备最后在线时间。若串口异常断开，线程会等待后重连，从而提高系统鲁棒性。

### 5.3.2 自动灌溉线程设计

自动灌溉线程按固定周期读取数据库中的灌溉策略，包括启用状态、土壤阈值、浇水时长和冷却时间。若策略启用且当前土壤湿度低于阈值，则线程向设备端发送“水泵开启”命令，等待指定时长后再发送“水泵关闭”命令，并在数据库中写入两条控制日志。线程还维护当前浇水状态、最近开始时间和最近结束时间，以供前端查询显示。该设计体现了边缘端根据最新感知数据进行本地决策的思想，是边缘计算特征的重要体现。

### 5.3.3 Web 接口设计

系统提供了较完整的 REST 风格接口。实时数据接口用于返回最新缓存值；历史数据接口支持按时间范围和数量查询历史传感器记录；历史 CSV 导出接口便于后期报表和分析；控制接口接收页面发来的命令并通过串口下发；灌溉策略接口支持查询与修改；登录注册接口负责用户认证；控制日志接口与设备状态接口便于后台管理。接口命名统一，参数结构清晰，能够支持前端页面和外部系统调用。

### 5.3.4 图像分析与 AI 助手设计

图像分析接口通过控制树莓派摄像头拍照，并调用 OpenCV 对图像进行颜色分割。程序在 HSV 空间中分别设定红色、绿色和粉色阈值，统计像素数量后选取得分最高的颜色，并映射为对应生长阶段。虽然该算法相对简单，但具备计算代价低、部署方便和结果直观等优点，适合作为毕业设计中的边缘视觉模块。

AI 助手接口则读取最新环境参数，拼接为提示词后送入轻量化本地模型，返回控制在百字以内的自然语言建议。这种设计体现了“规则外的补充性智能”，能够提升系统交互体验和展示效果。考虑到树莓派算力有限，后续可通过模型量化、知识规则融合或异步任务队列进一步优化响应速度。

## 5.4 数据库设计

数据库设计是本系统工程完整性的关键。系统设计了 7 张核心业务表：用户表 `users`、角色表 `roles`、用户角色关联表 `user_roles`、设备表 `devices`、环境数据表 `sensor_data`、控制日志表 `control_logs` 和灌溉策略表 `irrigation_policies`。其中，用户与角色是多对多关系，设备与历史数据、控制日志、灌溉策略均为一对多关系。

在具体实现中，环境数据表存储温度、湿度、光照、土壤湿度与时间戳，用于趋势分析；控制日志表记录执行器、动作、原始命令、是否成功与创建时间，用于审计追踪；灌溉策略表记录阈值、浇水时长、冷却时间与更新时间，用于支撑自动控制；设备表保存设备名称、描述和最近在线时间。系统还创建触发器，在控制日志插入后自动刷新对应设备的 `last_seen` 字段，使设备状态更新更及时。

### 5.4.1 权限认证设计

用户认证采用“用户名 + 密码 + 令牌”机制。注册时将密码进行哈希存储，登录成功后使用安全序列化方式生成令牌，并在后续请求中通过 Bearer Token 进行身份识别。系统默认将首个注册用户赋予管理员角色，从而实现初始系统管理。关键的灌溉策略修改与部分控制操作需要管理员身份或管理令牌，降低了误操作风险。

## 5.5 前端界面设计

前端页面主要由四个部分构成。实时监控页面用于展示温度、湿度、光照、土壤湿度和手势状态，并提供手动控制、拍照、AI 视觉分析和种植建议入口；历史页面基于 ECharts 绘制温湿度与光照土壤双图表，并支持时间区间查询和 CSV 导出；后台管理页面用于配置自动灌溉策略和查看控制日志；登录页面提供用户登录与注册功能。页面整体采用响应式布局，能够在电脑和移动端浏览器上适配显示。

## 6.3 性能测试与结果分析

### 6.3.1 实时性分析

设备端采样周期约为 1 s，边缘端页面通过定时请求实现实时刷新，整体适合中小型监测场景。由于系统采用本地串口传输和本地边缘计算，控制路径短，水泵等执行器能够较快响应页面指令。相较于完全依赖云平台的方案，本系统在局域网内具有更好的实时性。

### 6.3.2 稳定性分析

从软件结构上看，系统通过驱动降级、串口断线重试、异常捕获和线程分工提高了稳定性。设备端即便某一传感器异常，也不会导致整个程序立即崩溃；边缘端串口异常时会等待后重连；数据库采用 WAL 模式，有助于提升并发读写表现。对于毕业设计原型而言，这种稳定性设计具有较强工程意识。

### 6.3.3 存在问题分析

尽管系统已经实现了较完整功能，但仍存在一些可优化之处。第一，土壤湿度换算依赖经验阈值，需通过实测标定提高准确性。第二，视觉识别目前基于颜色阈值，受光照影响较大，若环境复杂可能误判。第三，轻量化大模型在树莓派上运行时延较高，不适合频繁调用。第四，数据库目前采用 SQLite，适合边缘原型，但在多用户并发和跨设备共享方面不如 MySQL 等数据库灵活。第五，项目文档中对本地模型版本存在表述不一致问题，最终定稿时应统一说明所使用的实际模型名称和版本。

总体而言，这些问题并不影响系统作为毕业设计作品的完整性，反而为后续优化与论文展望部分提供了较明确的方向。

# 7 总结和展望

## 7.1 总结

本文围绕藏红花培育场景，设计并实现了一套基于边缘计算的智能培育系统。系统以 STM32F411 为设备端核心，实现了温湿度、光照、土壤湿度和手势信息采集，以及 OLED 显示和执行器控制；以树莓派为边缘计算节点，实现了串口数据汇聚、数据库存储、自动灌溉、Web 服务、权限管理和图像分析；以浏览器为用户端，实现了实时监控、历史查询、后台管理、日志审计和登录认证等功能。系统具备较清晰的层次结构和较完整的业务闭环，较好地达到了本科毕业设计对软硬件协同、联网能力、数据库持久化、角色权限区分、界面展示和测试分析的综合要求。

从工程实践角度看，本系统的价值主要体现在三个方面。其一，验证了边缘计算在小型智能农业场景中的适用性：通过将实时数据处理与策略执行放在本地边缘端，可显著降低系统复杂度与云端依赖。其二，验证了使用低成本器件实现多功能协同的可行性：在预算有限条件下，仍然能够完成采集、显示、控制、视觉与智能建议等多模块集成。其三，验证了面向对象封装、数据库建模、接口设计与前端可视化在毕业设计中的综合应用价值，使作品不仅“能跑”，也具备一定的工程规范性。

## 7.2 展望

后续工作可从以下几个方向继续完善。

第一，在通信架构方面，可引入 MQTT Broker 或云服务器，将树莓派本地服务扩展为“边缘 + 云”协同模式，实现多终端同步、远程通知和跨地域访问。第二，在感知维度方面，可增加空气质量、CO₂、土壤温度、水位等传感器，提高对培育环境的综合描述能力。第三，在控制策略方面，可由简单阈值控制升级为模糊控制、规则引擎或基于历史数据的预测性控制，提高灌溉与补光策略的智能化水平。第四，在视觉分析方面，可采集更多藏红花样本图像，利用轻量级卷积神经网络或迁移学习模型替代纯颜色阈值法，提高复杂背景下的识别准确率。第五，在数据库方面，可在系统正式部署阶段将 SQLite 迁移到 MySQL，以支持更丰富的数据管理和多用户并发。第六，在硬件结构方面，可完成定制 PCB 与箱体集成，提高系统稳定性、美观度和可维护性。

## 7.3 个人总结与反思

通过本次毕业设计，我对嵌入式开发、边缘计算、数据库设计、Web 应用开发和软硬件协同有了更系统的理解。过去在学习过程中，我往往是将单片机、Python、数据库和前端页面分开理解，而本项目让我真正体会到一个完整系统从底层采集到上层展示之间需要大量接口衔接、异常处理和业务抽象。尤其是在树莓派与 STM32 协同开发、串口通信调试、数据库结构设计和前端页面联调过程中，我认识到工程实现不仅需要代码能力，更需要模块划分、调试思维和问题定位能力。

同时，我也意识到自己的不足。例如，硬件电路设计与器件选型经验仍需加强，很多模块在最初接线和调试时存在反复试错；对机器视觉和模型部署的理解还停留在基础应用层面，后续需要进一步学习更系统的方法；论文写作方面也需要在逻辑组织、图表表达和测试数据整理上持续提升。总体来说，这个项目不仅是一次毕业设计任务，更是一段将课堂知识落到真实系统中的综合训练过程，对今后的学习和工作都具有重要意义。
